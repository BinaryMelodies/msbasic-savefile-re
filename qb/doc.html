Binary save file formats for QuickBASIC and Visual Basic for MS-DOS

Preamble: This guide is not intended to be a comprehensive description of the syntactical structure of the QuickBASIC programming language. While the information should be sufficient to convert the binary form to a textual form (and hopefully the other way around), no attempt has been made to exclude invalid constructs. Furthermore, any semantical interpretation of the code, required to write an interpreter for example, is beyond the scope of this document, and is only given when relevant to parsing the binary format.

Overview

The present document attempts to describe the binary save format of QuickBASIC and Visual Basic for MS-DOS. The following versions have been studied:

  • [QB4.5] QuickBASIC 4.50
  • [QB7.0] QuickBASIC Extended 7.0

File layout

A QuickBASIC save file consists of 3 main parts: a header, a metadata table, and the procedure bodies and definitions.

The header

The first 3 bytes seem to determine what version of QuickBASIC or Visual Basic for DOS it was generated with. Here are the first 3 bytes as hexadecimal values and the length of the header:

Version First 3 bytes Size of header
QB4.0 FC 13 00 14 (0x0E) bytes
QB4.5 FC 00 01 28 (0x1C) bytes
QB7.0 FC 01 01 29 (0x1D) bytes
QB7.1 FC 02 01 29 (0x1D) bytes
VBDOS FC 08 01 at least 32 (0x20) bytes

For VBDOS forms, the header contains additional data, and the 16-bit little endian word at offset 0x14 stores how much additional bytes are contained in the header, beyond the default 32 bytes.

All offsets within the file are encoded relative to the end of the header.

The last 2 bytes of the header specify an offset, stored as a little endian 16-bit integer, from the end of the header to the start of the main procedure opcode stream.

The identifier table

Following the header is a series of metadata, including the identifier table. All identifiers used in the program have a corresponding entry in this region that they reference. An entry in the identifier table has the following format:

?? ?? <flags:U8> <length:U8> <identifier>

The exact meaning of the first two bytes is unknown, it appears to hold an offset, probably to the next identifier in a linked list.

The third byte contains some metadata information. Only bit 1 (0x02) is currently known: if clear, the identifier is encoded as a sequence of ASCII characters. If set, the identifier is encoded as an unsigned 16-bit little endian word, representing a line number. QuickBASIC stores certain line numbers (apparently below 65530) in this little endian format, all other line numbers are stored in their ASCII representation.

The fourth byte is the length of the identifier. For ASCII identifiers, this is the number of characters required to store them. For 16-bit labels stored as a 2-byte word, it is 2.

Procedure bodies and definitions

This part begins with a binary representation of the contents of the main module, then each SUB or FUNCTION definition follows. A SUB or FUNCTION procedure begins with a header. All three types of procedures continue with a sequence of 16-bit words describing the actual contents of the procedure, and terminated with a trailer part.

The header of a SUB or FUNCTION has the following format:

?? <length:U8> <name> ?? ?? <flags:U8>

The second byte contains the length of the name of the procedure, followed by the actual characters. Two unknown bytes follow, as well as a flags byte. Only bit 7 is currently known, it represents a STATIC procedure. Note that the STATIC keyword does not appear in the opcode stream.

All procedures are followed by an opcode stream representing the textual body of the procedure. The first two bytes encode the length of the opcode stream (not including the two bytes for the length), followed by a sequence of 16-bit little endian words.

All procedures are terminated with 16 bytes.

Encoding of procedure bodies

The way BASIC source code is encoded follows an abstract syntax tree format. This is in opposition to other versions of Microsoft BASIC (such as GW-BASIC or Microsoft QuickBASIC for the Macintosh) that instead store the source code as a compactified token stream.

The binary format is a sequence of 16-bit words that we will refer to as opcodes, interspersed with other data. The syntax tree can be understood to be constructed incrementally using these opcodes and data. Each opcode represents a step in the parsing process of the binary file. Some opcodes have parameters that follow them in the stream, such as the name of an identifier. Opcodes can also be preceded by parameters, similarly to how a stack machine would work. For example, an expression such as (1 + 2) * 3 would be stored as the sequence of opcodes corresponding to 1, 2, +, (), 3, *. The stack would then contain partial abstract syntax trees.

The format of opcode words has not changed significantly between BASIC versions, though there are some incompatibilities. The biggest change occured between versions 4.0 and 4.5: all the opcodes have been renumbered.

The most significant bits of an opcode can encode extra information. In QuickBASIC 4.0, only the 9 least significant bits determine the opcode type, whereas in QuickBASIC 4.5 and later, the 10 least significant bits are used.

Visual Basic for MS-DOS form layout

Visual Basic for MS-DOS comes with a visual form designer, and it can store it in a textual or binary format as well. The textual format is not documented to the author's knowledge, so it will also be explained in this document.

When saved in text format, the program code is prefixed with a hierarchical structured representation of the visual layout. The first line will be Version 1.00, followed by the layout of the main Form object. An object has the following layout:

BEGIN ControlType ControlName
        AttributeName1 = AttributeValue1
        AttributeName2 = AttributeValue2
⋮
END

An object may also have subobjects, embedded in it (for example, controls within a form). These will appear as a new BEGIN block, right after all the attributes of the containing object, indented. A simplified example for a form would be as follows:

Version 1.00
BEGIN Form MyForm
    Caption = "My Form"
    BEGIN CommandButton MyButton
        Caption = "My Button"
    END
END

' Here comes the program code

Controls and attributes

Visual Basic for MS-DOS supports the following control types: CheckBox, ComboBox, CommandButton, DirListBox, DriveListBox, FileListBox, Form, Frame, HScrollBar, MDIForm, Menu, Label, ListBox, OptionButton, PictureBox, TextBox, Timer, VScrollBar. A form is always either Form or MDIForm, and these can only appear at the top level.

We will not document all the valid and invalid options here, and will instead focus on what can appear in a textual source code.

Attributes can take several types of values on the right side of the equality sign. Strings are enclosed within " symbols, and any " character inside is represented by a doubled "" sequence. Integers always appear in decimal form, and they can be prefixed by a - sign. Booleans are represented by 0 for False and -1 for True, as the convention for Visual Basic. Some attributes can have values that appear as Char(value) or QBColor(value) where value is a non-negative decimal literal.

The only exception to these types is the Shortcut attribute, whose value is a symbolic expression describing a key combination. The general syntax is [shift] [control] (letter | function key), where shift is represented via the + symbol, control via ^, letter as a single capital letter, and function key as an F-number enclosed in braces: {Fn} where n is the function key number.

Visual Basic for MS-DOS saves all attribute values, except for Index and Shortcut, if they are absent. Also note that MDIForm forms do not have a WindowState field, as it is implicitly set to 2 (maximized).

Binary file layout

When stored in binary format, the form layout is stored as part of the header. Unlike all versions of QuickBASIC, the header is now of a dynamic size, and 16-bit word at offset 0x14 gives the additional bytes of information in the form. The layout starts at offset 0x16, and all strings are represented as displacements from this point.

The overall format of the layout is: 10 bytes of metadata, the form/control records, the string table, the form/control names, and two terminating bytes.

Metadata

The 10 bytes of metadata are as follows:

<flags:U8> ?? ?? ?? ?? ?? <offset to names:U16> <total length of records:U16>

When bit 0x04 in flags is set, the form is an MDIForm instead of a Form. The two types are only distinguished by this flag.

Form/control records

There can be an arbitrary number of records, and the size of each record depends on the type of the control. Attributes are represented here as the following types:

BOOLEAN
A bit in a bitfield, either set or clear
INTEGER
An integer, between 0 and 32767
SIGNED
An integer, between -32768 and 32767
UNSIGNED
An integer, between 0 and 65535
CHAR
An integer, represented in the source code using the Char(value) syntax
QBCOLOR
An integer, represented in the source code using the QBColor(value) syntax
STRING
A string, represented as an offset into the string table

The field offset to parent record is used to describe the control hierarchy. Each control can appear inside another control or form, and this field stores the offset from 0x16 to the start of the parent record.

For Forms, Top, Left, Height and Width stores the values for the normal state, i. e. when the window is neither minimized nor maximized. For Menus, Caption stores the Shortcut field as well, separated from the caption by a tab character. The Shortcut field is stored in the format it displays at runtime, so a capital letter or Fn for a function key, prefixed by Shift+ and Ctrl+ if needed.

String table

The string table is a sequence of strings, each prefixed by a 16-bit word to signify the length. Each entry is addressed as an offset relative to 0x16. Visual Basic for MS-DOS stores strings separately for each instance they appear as, even if it results in duplication.

Name table

The name table is a sequence of strings, each prefixed by an 8-bit word to signifiy the length. Each entry is addressed as a 0-based index in the list. Two controls cannot share the same name, except if they belong in the same control array, in which case their names appear only once in the name table.

The name table is followed by two bytes of unknown value, terminating the additional bytes for the header.
